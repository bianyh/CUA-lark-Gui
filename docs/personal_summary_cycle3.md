# 第三周期（4.29-5.03）

## 一、核心产出

本周期最主要的成果，是把 CUA-Lark 从“能观察飞书窗口、能做基础动作”的状态，推进到了真实 API + 真实飞书界面下可完成端到端任务的阶段。

这次重点不是继续堆脚本步骤，而是修正了 planner 的工作方式：`metadata.scripted_actions` 不再是必须顺序执行的固定计划，而是作为模型规划时的参考 hints。真实运行时，系统每一步都会根据当前截图、历史动作、进度评估和失败反思重新决定下一步。这个改动让 plan 智能体更接近真正的 Computer-Use Agent，而不是 YAML 脚本播放器。

本周期另一个关键成果，是完成了真实飞书 IM 用例 `im_send_message` 的闭环调试。最终系统可以在真实飞书窗口中进入“测试群”，输入并发送 `Hello World`，然后由模型语义校验确认发送成功。最新成功报告显示，真实任务在 `57.42s` 内完成，动作路径为：

```text
hotkey -> click -> type_text -> hotkey
```

其中包括真实快捷键、真实坐标点击、真实文本输入、真实发送和真实报告产出。

这次也把 OCR 策略做了调整。PaddleOCR 在本机环境里返回 numpy array，原解析逻辑会因为数组布尔判断直接崩溃。考虑到当前目标是 vision-first GUI agent，我把默认 OCR 后端改成 `none`，让系统主要依赖截图和大模型视觉理解；同时保留 PaddleOCR 的可选入口，并为 OCR 异常加了兜底，避免 OCR 问题拖垮主流程。

## 三、过程复盘与沉淀

### 1. 主要完成了什么？

这周期主要完成了四件事。

第一，是把 planner 从“预设脚本优先”改成“动态下一步规划”。之前只要 case 里有 `scripted_actions`，planner 就会按索引一个个执行，模型基本没有机会根据当前界面改变策略。现在真实模式下，脚本动作只作为参考，模型会结合当前 observation、history、progress、reflection 重新选择下一步。

第二，是打通真实飞书 IM 场景。调试过程中先后暴露出几个真实问题：模型返回动作字段不稳定、坐标没有真正落到屏幕、中文/英文输入没有聚焦输入框、PaddleOCR 抢在模型前崩溃。每个问题都通过日志、run.json 和 timeline 截图逐步定位，最后跑通了真实发送消息任务。

第三，是增强 Windows 执行器。模型看到的是飞书窗口截图坐标，而 `pyautogui` 需要屏幕绝对坐标，所以执行器现在会把窗口内坐标转换成屏幕坐标。文本输入也不再依赖逐字符键盘输入，而是走 `pyperclip` 剪贴板粘贴；如果模型只给 `type_text` 而没有坐标，执行器会先点击飞书窗口右下方消息输入框区域，再粘贴文本。

第四，是完善可调试性。真实运行时，控制台会输出观察、规划、执行、状态等待、进度评估、反思和最终验证；报告里会记录动作、截图、执行元数据、进度和最终断言。通过这些信息，可以比较直接地复盘一次 GUI agent 为什么失败、下一轮应该改哪里。

### 2. 遇到的问题和解决方式

第一个问题是模型 action schema 和代码 schema 不一致。

模型经常返回：

```json
{"type": "click", "x": 115, "y": 85}
```

而原代码只认：

```json
{"action_type": "click", "coordinates": [115, 85]}
```

结果这些动作被解析成默认 `wait`，所以看起来像 `ctrl+k` 或点击没有效果，实际是 executor 根本没执行对应动作。后来我把 `ActionStep.from_dict()` 做成更宽容的解析：支持 `type` 别名，支持 `x/y` 合成坐标，支持 `"ctrl+k"` 拆成 `["ctrl", "k"]`，也支持 `input/text` 映射成 `type_text`。

第二个问题是模型坐标点击和真实桌面坐标不是同一套坐标系。

飞书截图是窗口区域，模型输出的是窗口内像素坐标；但鼠标点击需要屏幕绝对坐标。这个问题通过执行器统一转换解决：先读取飞书窗口 `left/top/width/height`，再把窗口内坐标加上窗口左上角偏移，得到屏幕坐标。

第三个问题是文本输入没有落到输入框。

模型在进入“测试群”后能判断要输入 `Hello World`，但如果输入框没有焦点，直接粘贴不会生效。后来在 `type_text` 无坐标时增加了默认输入框聚焦策略：先点击飞书窗口右下方消息输入区，再粘贴文本。这个改完之后，真实运行中 `Hello World` 能出现在输入框里，随后模型按 Enter 发送成功。

第四个问题是 OCR 依赖不稳定。

PaddleOCR 返回 numpy array 后，原来的 `a or b or c` 写法会触发 `ValueError: The truth value of an array with more than one element is ambiguous`。这个问题一方面通过安全解析修掉，另一方面把默认 OCR 后端改成 `none`，让主链路回到视觉大模型理解截图，不把 OCR 作为必需前提。

### 3. 可复用沉淀

这周期沉淀下来的东西，后面做其他飞书场景可以直接复用。

- 动态 planner 模式：脚本步骤只做 hints，不再作为强制计划。
- 模型 action 宽容解析：兼容 `type/x/y/hotkey/text` 等常见输出格式。
- 窗口截图坐标到屏幕坐标的统一映射。
- 文本输入策略：优先剪贴板粘贴，必要时先点击默认输入区域。
- OCR 非必需化：OCR 可以辅助，但不能成为主流程单点故障。
- 真实调试方法：先看控制台，再看 `run.json`，最后对照 timeline 截图定位问题。

## 四、本周随手记

- GUI agent 的问题经常不是“模型不会做”，而是模型说的话和执行器能理解的动作格式不一致。
- 对真实桌面自动化来说，报告和截图不是附属品，而是调试主工具。没有 timeline，很难判断到底是模型想错了、解析错了，还是鼠标点错了。
- OCR 在这个阶段应该降级为可选增强能力。只要大模型能直接看图，主链路就不应该被 OCR 依赖阻断。
- `type_text` 这类动作不能只考虑“怎么输入文字”，还要考虑“输入目标是否已经聚焦”。这比单纯调用键盘 API 更关键。
- 坐标点击能力已经可用，但后续还需要继续加强模型 prompt 和停滞检测，减少“重复点击同一输入框”这类低效行为。
